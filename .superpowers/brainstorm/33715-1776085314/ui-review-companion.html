Findings
High

Project detail rows are hidden before they intersect

The detail page starts rows in a hidden visual state and waits for IntersectionObserver to reveal them. Because the hidden rows still take up normal height, the design language creates empty narrative gaps instead of progressive disclosure.

  • Good motion should decorate reading flow, not block it.
  • This is especially risky on mobile where viewport depth is tight.
components/project-detail.tsx:99, 568-580 components/project-detail.module.css:317-331
High

Mobile header preserves desktop information density

The header keeps logo, four nav links, and two utility toggles exposed at the smallest breakpoint. The CSS wraps the system rather than redesigning it, so mobile pays a heavy vertical tax before content even begins.

  • Good mobile chrome compresses choices and protects first-screen content.
  • The current approach treats all header elements as equally important.
components/site-header.module.css:10-15, 71-84
Medium

Interior pages are systemically left-heavy

Shared globals pin titles, subtitles, copy, and contact content into the left half of the 12-column system. The right side is usually unused, so the composition reads incomplete rather than intentional.

app/globals.css:278-289 app/globals.css:356-389
Medium

Interaction state language is inconsistent

Active nav uses weight only, project tabs use a thin underline, the grid utility fills its background, and the theme toggle only swaps icon visibility. The shell lacks one coherent vocabulary for state.

Medium

About tech-stack hierarchy is only semantic, not visual

Categories exist in code and ARIA labels, but users see three chip rows without visible labels. Color alone is doing categorization work that should be reinforced by structure.

Medium

Mobile project-card reading order is awkward

The arrow action lives before the description in DOM and becomes a full row at the mobile breakpoint. That makes the card scan as title, role, arrow, then description.
